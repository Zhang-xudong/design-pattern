1.最基本的数据访问程序

在日常的开发中我们肯定会和数据库打交道，现在我们假设要访问的数据库是SQL Server,并且进行两个操作，一个是插入，一个是查找。对应有两个类，一个是实体类User,一个是用于操作SQL Server中User表的类SqlServerUser。

class User{
    private int id;
    private String name;

    public void setId(int id){
        this.id = id;
    }
    public int getId(){
        return this.id;
    }

    public void setName(String name){
        this.name = name;
    }
    public String getName(){
        return this.name;
    }
}

class SqlServerUser {
    public void insert(User user){
        System.out.println("在SQL Server中给User表增加一条记录")
    }

    public User getUser(int id){
        System.out.println("在SQL Server中根据id的到User表中的一条记录")
            return null;
    }
}

客户端代码：

static void main(String[] args){
    User user = new User();
    SqlServerUser su = new SqlServerUser();

    su.insert(user);
    User user2 = su.getUser(1);
}

假如现在我们要换数据库，那么上面的代码就不能用了，因为SqlServerUser su = new SqlServerUser()使得su这个对象框死在SQL Server上了，如果这里是灵活的，或者说是多态的，那么在执行su.insert(user)和su.getUser(1)的时候就不需要考虑到底现在是用SQL Server还是其他数据库了（比如Acess数据库）。我们现在用工厂方法模式来封装一下new SqlServerUser()所造成的变化。工厂方法模式就是定义一个用于创造对象的接口，让子类决定实例化哪一个类。

2.用了工厂方法模式的数据访问程序

我们先来看一下UML图：

IUser接口，用于客户端访问，解除与具体数据库访问的耦合：

interface IUser {
    void insert(User user);
    User getUser(int id);
}

SqlServerUser类，用于访问SQL Server的User表

class SqlServerUser implements IUser{
    public void insert(User user){
        System.out.println("在SQL Server中给User表增加一条记录")
    }

    public User getUser(int id){
        System.out.println("在SQL Server中根据id的到User表中的一条记录")
            return null;
    }
}

AccessUser类，用于访问Access的User表

class AccessUser implements IUser{
    public void insert(User user){
        System.out.println("在Access中给User表增加一条记录")
    }

    public User getUser(int id){
        System.out.println("在Access中根据id的到User表中的一条记录")
            return null;
    }
}

IFactory接口，定义一个创建访问User表对象的抽象的工厂接口

interface IFactory{
    IUser createUser();
}

SqlServerFactory类，实现IFactory接口，实例化SqlServerUser：

class SqlServerFactory implements IFactory{
    public IUser createUser(){
        return new SqlServerUser();
    }
}

AccessFactory类，实现IFactory接口，实例化AccessUser：

class AccessFactory implements IFactory{
    public IUser createUser(){
        return new AccessUser();
    }
}

客户端代码：

static void main(String[] args){
    User user = new User();
    //若要改成Access数据库，只需要将该句改成
    // IFactory factory = new AccessFactory()
    IFactory factory = new SqlServerFactory();

    IUser iu = factory.createUser();
    iu.insert(user);
    User user2 = iu.getUser(1);
}

如上，现在如果要换数据库，只需要把new SqtServerFactory(）改成new AccessFactory()，此时由多态的关系，使得声明lUser接口的对象iu事先根木不知道是在访问哪个数据库，却可以在运行时很 好的完成工作，这就是所谓的业务逻辑与数据访问的解耦。

3.用了抽象工厂模式的数据访问程序

先看UML图：

当然，数据库中不止一张表，所以我们除了User表又另外添加了一张部门表Department。

interface IDepartment{
    void insert(Department department);
    Department getDepartment(int id);
}

SqlSeverDepartment implements IDepartment{
    public void insert(Department department){
        System.out.println("在SQL Server中给Department表增加一条记录");
    }
    public Department getDepartment(int id){
        System.out.println("在SQL Server中根据id的到Department表中的一条记录");
    }
}

AcessDepartment implements IDepartment{
    public void insert(Department department){
        System.out.println("在Access中给Department表增加一条记录");
    }
    public Department getDepartment(int id){
        System.out.println("在Access中根据id的到Department表中的一条记录");
    }
}

IFactory接口

interface IFactory{
    IUser createUser();
    //增加的接口方法
    IDepartment createDepartment();
}

SqlServerFactory类

class SqlServerFactory implements IFactory{
    public IUser createUser(){
        return new SqlServerUser();
    }

    //增加SqlServerDepartment工厂
    public IDepartment createDepartment(){
        return new SqlServerDepartment();
    }
}

AccessFactory类

class AccessFactory implements IFactory{
    public IUser createUser(){
        return new SqlServerUser();
    }

    //增加SqlServerDepartment工厂
    public IDepartment createDepartment(){
        return new SqlServerDepartment();
    }
}

客户端代码：

static void main(String[] args){
    User user = new User();
    Department department = new Department();

    //只需确定实例化那个数据库访问对象给factory
    //IFactory factory = new SqlServerFactory();
    IFactory factory = new AccessFactory();
    //则此时已与具体的数据库访问解除了依赖
    IUser iu = factory.createUser();
    iu.insert(user);
    iu.getUser(1);

    //则此时已与具体的数据库访问解除了依赖
    IDepartment id = factory.createDepartment();
    id.insert(department);
    id.getDepartment(1);
}

结果显示如下：

在Access中给User表增加一条记录
在Access中根据id的到User表中的一条记录
在Access中给Department表增加一条记录
在Access中根据id的到Department表中的一条记录

像上面这样，当产品比较单一，比如只有一个User的时候，我们可以用工厂方法模式。而涉及到多个产品系类的问题时，抽象工厂模式就更加合适。上面的情况，在我们又引入了Department后，采用的就是抽象工厂模式。

4.抽象工厂模式

抽象工厂模式：提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类。

AbstractProductA和AbstractProductB是两个抽象产品，之所以为抽象，是因为它们都有可能有两种不同的实现，就刚才‘的例子来说就是User和Department, 而ProductA1、 ProductA2和ProductB1、 ProductB2就是对两个抽象产品的具体分类的实现，比如ProductA1可以理解为是SqlserverUser，而 ProductB1是AccessUser。

这么说，IFactorv是一个抽象工厂接口，它里面应该包含所有的产品创建的抽象方法。而 ConcreteFactory1和ConcreteFactory2就是具体的工厂了。就像SqlserverFactory和AccessFactory一样。通常是在运行时刻再创建一个ConcreteFactory类的实例．这个具体的工厂再创 建具有特定实现的产品对象，也就是说，为创建不同的产品对象．客户端应使用不同的具体工厂。

5.抽象工厂模式的优缺点

通过上面的分析介绍会发现，抽象工程模式最大的好处就是易于交换产品系列，由于具体工厂类，例如IFactory factory = new AccessFactory()，在一个应用中只需要在初始化的时候出现一次，这就使得改变一个应用的具体工厂变得非常容易，她只需要改变具体工厂即可使用不同的产品配置。我们的实际应用不能防止需求的变更，那么我们的理想便是让改动变得最小，现在如果你要改变数据库的访问，我们只需要更改具体工厂就可以做到。第二大好处就是，它让具体的创造实例过程与客户端分离，客户端是通过它们的抽象接口来操纵实例，产品的具体类名也被具体工厂的实现分离，不会出现在客户端代码中。事实上，我们刚才写的代码中，客户端所认识的只有IUser和IDpartment,至于它使用SQL Server还是Access来实现的就不知道了。

每一种设计模式都有其局限性，抽象工厂也不例外，我们来看看它有什么缺点。假如应用中要增加一张项目表Project,我们需要改动哪些地方？最起码要增加三个类：IProject、SqlServerProject、AccessProject,还需要更改IFactory、SqlServerFactory、AccessFactory才可以完全实现。是不是觉得有点糟糕？我们的客户端很可能不止一个，有很多地方可能要用到IUser和IDepartment,而这样的设计，其实在每个类的开始（客户端的开始）都需要声明IFactory factory = new SqlServerFactory()，如果我有100个调用数据库访问的类，是不是就要更改100次IFactory factory = new SqlServerFactory()这样的代码才行？这不能解决我们要更改数据库访问时，改动一处就完全更改的要求。编程是门艺术，这样大批量的改动，显然是非常丑陋的做法。那有没有什么方法改进呢？我们下面着手一步步改一下试试看。

6.抽象工厂模式的改造

6.1用简单工厂来改进抽象工厂

为了避免上文中所提到的抽象工厂模式的缺点，我们尝试用简单工厂模式来改造一下程序。我们去除IFactory、SqlServerFactory和AccessFactory三个工厂类,取而代之的是DataAccess类，用一个简单工厂来实现。

class DataAccess{
    private static final String db = "SqlServer";
    //private static final String db = "Access";

    //由于db事先的设置，所以此处可以根据选择实例化出相应的对象
    public static IUser createUser(){
        IUser result = null;
        switch(db){
            case "SqlServer":
                result = new SqlServerUser();
                break;
            case "Access":
                result = new AccessUser();
                break;
        }
        return result;
    }

    //由于db事先的设置，所以此处可以根据选择实例化出相应的对象
    public static IDepartment createDepartment(){
        IDepartment result = null;
        switch(db){
            case "SqlServer":
                result = new SqlServerDepartment();
                break;
            case "Access":
                result = new AccessDepartment();
                break;
        }
        return result;
    }
}

客户端代码：

static void main(String[] args){
    User user = new User();
    Department department = new Department();

    //直接得到数据库访问实例，而不存在任何依赖
    IUser iu = DataAccess.createUser();
    iu.insert(user);
    iu.getUser(1);

    IDepartment id = DataAccess.createDepartment();
    id.insert(department);
    id.getDepartment(1);
}

像上面这样，相比抽象工厂模式与其用那么多工厂类，不如直接用一个简单工厂模式来实现。我们抛弃了IFactory、SqlServerFactory和AccessFactory三个工厂类,取而代之的是DataAccess类，由于实现设置了db的值（SqlServer或Access）,所以简单工厂不需要传入输入参数，这样就在客户端就只需要DataAccess.createUser()和DataAccess.createDepartment()来生成具体的数据库访问实例，客户端没有出现任何一个SqlServer或Access的字样，达到了解耦的目的。

那如果现在需要增加Oracle数据库的访问，本来抽象工厂模式只增加一个OracleFactory工厂类就可以了，现在就比较麻烦了，需要在DataAccess类中每个方法的switch中增加对应的case语句。

6.2用反射+抽象工厂的数据访问程序

像上面那种情况下，我们要考虑的就是可不可以不在程序里面写明“如果是SqlServer就去实例化SQLServer数据库相关的类，如果是Access就去实例化Access相关的类”这样的语句。而是根据字符串db的值去某个地方找应该要实例化的类是哪一个。这样我们的switch就可以和它说再见了。那什么叫“去某个地方找应该要实例化的类是哪一个？”，这就是我们熟悉的一种编程方式——依赖注入（Dependency Injection）。那我们如何运用这种依赖注入的编程方式去解决switch的问题呢？本来依赖注入是需要提供专门的IoC容器提供，比如Spring中的IoC容器，显然当前这个程序不需要这么复杂，你只需要了解java中的“反射”技术就可以了。运用了反射技术就可以帮助我们克服抽象工厂模式的先天不足了。

有了反射，我们获取实例就可以用下面两种方式：

//常规写法
IUser result = new SqlServerUser();

//反射写法
IUser result = (IUser)Class.forName("要实例化类的全限定类名").newInstance();

上面两种方式的实例化效果是一样的，但它们的区别在那呢？常规写法是写明了要实例化SqlServerUser类，反射的写法也是写明了要实例化SqlServerUser对象（全限定类名）。但是常规方法不能够更灵活的更换为AccessUser,因为都是 事先编译好的代码，但反射方式就不一样， 反射中的全限定类名是字符串，这样它就可以用变量来处理，也可以根据需要进行更换。总结起来就是，常规的写法中实例化是写死在程序中的，而反射这种方式就可以利用字符串变量来实例化对象，而变量是可以更换的。准确的说，就是将程序由编译时转换为运行时。由于Class.forName("要实例化类的全限定类名")中的字符串是可以写成变量的，而变量的值到底是SQLServer还是Access,完全可以由事先的那个db变量来决定，所以就去除了switch判断的麻烦。

代码结构图：

class DataAccess{
    //包名
    private static  String packageName = "包名";
    //数据库类名，可以替换为Access
    private static  String db = "SqlServer";

    public static IUser createUser(){
        String className = packageName + "." + db + "User";
        IUser iu = (IUser)Class.forName(className).newInstance();
        return iu;
    }

    public static IDepartment createDepartment(){
        String className = packageName + "." + db + "User";
        IDepartment id = (IDepartment)Class.forName(className).newInstance();
        return id;
    }
}

现在如果我们增加了Oracle数据库访问，相关的类的增加是不可避免的，这点无论我们用任何方法都是解决不了的，不过这叫扩展，开放-封闭性原则告诉我们，对于扩展，我们开放，但对于修改，我们尽量要关闭。就不目前而言，我们只需要更改private static String db = "SqlServer"为private static String db = "Oracle"，也就意味着

(IUser)Class.forName(className).newInstance();这一句发生了变化。这样的结果就是DataAccess.createUser()本来得到的是SqlServer对象实例，现在变成了Oracle的实例了。

那假如我们现在要增加Project产品时，改怎么做呢？只需要增加三个与Project相关的类，再修改DataAccess，在其中增加一个public static IProject createProject()方法就可以了。

其实上面的方式还有一点缺陷，因为在更换数据库访问时，我们还是需要去改程序（改db这个字符串的值）重新编译，如果可以真正的不改程序才是真正的符合开放-封闭原则。

6.3用反射+配置文件实现数据访问程序

针对上面提到的问题，我们可以通过读取配置文件来给db赋值，在配置文件中写明时SqlServer还是Oracle，还是Access,这样就不用去修改DataAccess类了。

添加一个App.config文件，内容如下：

<? xml version="1.0" encoding="utf-8" ?>
<configuration>
    <appSettings>
        <add key = "DB" value="SqlServer"/>
    </appSettings>
</configuration>

然后我们通过读取这个配置文件并解析拿到DB对应的value，然后赋值给DataAccess中的变量db就可以了。这样，我们就通过反射+抽象工厂模式解决了数据访问时的可扩展和可维护问题。从这个角度来讲，所有在用简单工厂模式的地方，都可以考虑用反射技术来去除switch或if，解除分支判断带来的耦合。switch或则if是程序里的好东西，但在应对变化上却先得过于臃肿。反射技术却可以很好得解决它们应对变化，难以维护和扩展得缺点。
