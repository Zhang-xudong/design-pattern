1.简单工厂模式实现

承接上一节所讲的简单工厂模式，作为对比看一下工厂方法模式。同样还是计算器的实现看一下两者的UML类图：

工厂类是这样写的：

public class OperationFactory {
    public static Operation createOperation(String operate){
        Operation oper = null;
        switch (operate){
            case "+":
                oper = new AddOperation();
                break;
            case "-":
                oper = new SubOperation();
                break;
            case "*":
                oper = new MulOperation();
                break;
            case "/":
                oper = DivOperation();
                break;
        }
        return oper;
    }
}

客户端代码：

Operation oper;
oper = OperationFactory.createOperation("+");
oper.numA = 1;
oper.numB = 2;
double result = oper.getResult();

2.工厂方法模式实现

如果换成用工厂方法模式来实现呢？它的结构图如下：

先创建一个接口：

interface IFactory{
    Operation CreateOperation();
}

然后加减乘除各自创建一个具体的工厂去实现这个接口：

//加法类工厂
class AddFactory extends IFactory{
    public Operation CreateOperation(){
        return new AddOperation();
    }
}

//减法类工厂
class SubFactory extends IFactory{
    public Operation CreateOperation(){
        return new SubOperation();
    }
}

//乘法类工厂
class MulFactory extends IFactory{
    public Operation CreateOperation(){
        return new MulOperation();
    }
}

//除法类工厂
class DivFactory extends IFactory{
    public Operation CreateOperation(){
        return new DivOperation();
    }
}

客户端实现代码：

IFactory operFactory = new AddFactory();
Operation oper = operFactory.createOperation();
oper.numberA = 1;
oper.numberB = 2;
double result = oper.getResult();

3.简单工厂 VS 工厂方法

对比简单工厂模式，假如现在需要增加其他运算，比如开根号运算。在简单工厂里，我们要先去增加一个“开根号”的运算类，然后去改工厂方法，在其中加‘case'语句来做判断。现在用了工厂方法模式，除了添加加运算类，还需要添加一个对应工厂类，之后还要修改客户端的代码。这样大体一看，相比简单工厂模式，工厂方法模式不但没有减化难度，反而增加了很多类和方法，把复杂性增加了。既然这样，为什么还要搞个工厂方法模式，就踏实的用简单工厂模式不行吗？

简单工厂模式的最大优点在于工厂类中包含了必要的逻辑判断，根据客户端的选择条件动态实例化相关的类，对于客户端来说，去除了与具体产品的依赖。就像你的计算器，让客户端不用管该用哪个类的实例，只需要把‘+’给工厂，工厂自动就给出了相应的实例，客户端只要去做运算就可以了，不同的实例会实现不同的运算。但问题也就在这里，如果要加个‘开根号’的功能，我们是一定需要给运算工厂类的方法里 加‘case’的分支条件的，修改原有的类？这可不是好办法，这就等于说，我们不但对扩展开放了，对修改也开放了。这就违背了“开放-封闭”原则。于是工厂方法模式就出现了。

工厂方法模式：定义一个用于创建对象的接口，让子类决定实例化哪一个类。工厂方法使一个类的实例化延迟到其子类。

我们把上面的分析过程总结一下：在简单工厂模式中，既然这个工厂类与分支耦合，那我们就对他（工厂类）下手，根据依赖倒转原则，我们把工厂类抽象出一个接口，这个接口只有一个方法。然后，所有要生产具体类（具体的加减乘除运算类）的工厂，就去实现这个接口，这样一个简单工厂模式的工厂类，就变成了一个工厂抽象接口和多个具体的生成对象的工厂。于是当我们要增加“求M数的N次方”的功能时，就不需要更改原有的工厂类了，只需要增加此功能的运算类和和相应的工厂类就可以了。

如上，这样整个工厂和产品体系其实都没有修改的变化，而只有扩展的变化，这就完全符合了“开放-封闭”的原则精神。从这个角度讲，工厂方法模式确实要比简单工厂模式强大。

通过上面的一列分析，其实我们仔细观察就会发现，工厂方法模式实现时，客户端需要决定具体实例化哪一个工厂来实现运算类，选择判断的问题还是存在的，也就是说，工厂方法模式把简单工厂模式的内部逻辑判断转移到了客户端代码来进行，你想要家功能，本来是改工厂类的，而现在是修改客户端！

4.雷锋工厂

做这样一个假设：你们班级里有一个学雷锋爱做好人好事的同学叫薛磊风，他每个周末都会到学校周边一些孤寡老人的家中帮老人们做一些洗衣洗地、买米买面的日常家务。可是不久前他生病了，不能继续到老人家里帮忙了，所以他想在生病期间把继续照顾老人的任务委托给班里的其他同学。雷锋是众人皆知的做好人好事的模范，你们班里的这位同学以学习雷锋的名义做好事，而你们现在需要代替他去做好事，这其实就是典型的工厂模式应用了。我们通过代码来看一下，首先：雷锋类，拥有扫地、洗衣、买米能方法：

//雷锋
class LeiFeng{
    public void sweep() {
        System.out.println("扫地");
    }

    public void wash() {
        System.out.println("洗衣");
    }

    public void buyRice(){
        System.out.println("买米");
    }
}

学习雷锋的大学生继承雷锋类

//学习雷锋的大学生
class Undergratuate extends LeiFeng{

}

然后客户端代码如下：

LeiFeng xueleifen = new Undergratuate();

xueleifeng.sweep();
xueleifeng.wash();
xueleifeng.buyRich();

现在假设有三个人要替代生病的薛磊风去照顾老人，那代码改怎么写？看看如下的怎么样：

LeiFeng student1 = new Undergratuate();
student1.sweep();

LeiFeng student2 = new Undergratuate();
student2.wash();

LeiFeng student3 = new Undergratuate();
student3.buyRice();

假如你们现在要毕业了，而帮组老人是个长期的工作，所以“社区志愿者”更合适，此时上面的写法就不太合适了，因为我们需要更改多个实例化的地方。其实老人不需要直到是谁来做的好事，她只需直到是学雷锋的人来帮忙就可以了。所以还需增加一个继承“雷锋”类的“社区志愿者”类。

//社区志愿者
class Volunteer extends LeiFeng{

}

再写简单工厂类：

class SimpleFactory{
    public static LeiFeng createLeiFeng(String type) {
        LeiFeng result = null;
        switch(type){
            case "学习雷锋的大学生":
                result = new Undergratuate();
                break;
            case "社区志愿者":
                result = new Volunteer();
                break;
        }
        return result;
    }
}

客户端代码，如果要换只需把“学习雷锋的大学生”换为“社区志愿者”即可：

//简单工厂模式
LeiFeng student1 = SimpleFactory.createLeiFeng("学习雷锋的大学生");
student1.sweep();

LeiFeng student2 = SimpleFactory.createLeiFeng("学习雷锋的大学生");
student2.wash();

LeiFeng student3 =SimpleFactory.createLeiFeng("学习雷锋的大学生");
student3.buyRice();

此时你会发现，你需要在任何实例化的时候写出这个工厂的代码，这样有重复，我们试着再用工厂方法模式写一遍。这样，我们就需要一个雷锋工厂：

//雷锋工厂
interface IFactory {
    LeiFeng createLeiFeng();
}

//学雷锋的大学生工厂
class UndergratuateFactory implements IFactory {
    LeiFeng createLeiFeng() {
        return new Undergratuate();
    }
}

//社区志愿者工厂
class VolunteerFactory implements IFactory {
    LeiFeng createLeiFeng() {
        return new Volunteer();
    }
}

客户端调用的时候就可以像如下这样写了：

//工厂方法模式
IFactory factory = new UndergratuateFactory();
LeiFeng student = factory.createLeiFeng();

student.sweep();
student.wash();
student.buyRice();

综合上面的过程我们可以发现，工厂方法克服了简单工厂违背开放-封闭原则的缺点，又保持了封装对象创建过程的优点。工厂方法和简单工厂都是几种封装了对象的创建，使得要更换对象时，不需要做大量的改动就可以实现，降低了客户程序和产品对象的耦合。工厂方法模式是简单工厂模式的进一步抽象和推广。由于使用了多态性，工厂方法模式保持了简单工厂模式的优点，而且克服了它的缺点。但相对而言，工厂方法模式的缺点就是，由于每增加一个产品，就需要增加一个与产品对应的工厂类，增加了额外的开发量。
